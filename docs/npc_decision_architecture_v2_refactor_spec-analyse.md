## 1. Резюме

Спецификация грамотно диагностирует реальную проблему и предлагает архитектурно верный путь её решения. Предложенная PEIA-модель (Perceive → Evaluate → Intend → Act) широко применяется в игровом AI и хорошо масштабируется. Стратегия поэтапной миграции вместо big bang rewrite — правильная. Документ является хорошей архитектурной точкой отсчёта.

Вместе с тем документ описывает **что** строить, но не **как именно** — конкретные Python-типы, формулы, правила приоритизации и интеграционные контракты нуждаются в проработке до начала реализации. Ряд аспектов несёт **технические риски**, которые важно учесть на старте.

### 2.2. Ключевые проблемы текущего кода

Главная функция `_run_bot_action_inner` (~560 строк) реализует всю логику принятия решений как каскад условных проверок:

```python
# В текущем коде порядок ЖЁСТКО ВСТРОЕН:
1. Emission escape
2. Emission shelter
3. Global goal achieved → leave zone
4. Arrival commitment (pickup)
5. EMERGENCY: Heal
6. EMERGENCY: Eat
7. EMERGENCY: Drink
8. Sell on arrival
9. Equipment maintenance (weapon → armor → ammo)
10. Sleep
11. Gather resources / pursue global goal
```

Проблема: добавить **социальный intent** в этот каскад практически невозможно без переписывания. Каждый if-блок содержит логику из нескольких доменов: принятие решения, запись памяти, навигация, взаимодействие с инвентарём — всё перемешано.

Также нет структурной разницы между «я решил что-то сделать» и «я выполняю действие» — это два разных события, которые сейчас происходят в одном месте.

## 3. Оценка архитектурных предложений

### 3.1. Слой 1: AgentContext ✅ Правильная идея, хорошо выполнима

**Проблема решается корректно.** Сейчас каждая функция принимает `agent`, `state`, `world_turn` отдельно и делает свои обращения к `state["locations"]`, `state["agents"]` и пр. `AgentContext` нормализует это.

**Что важно учесть при реализации:**

- `AgentContext` должен быть вычислен **один раз** на тик для каждого агента — не при каждом вызове вспомогательной функции. Иначе O(N²) обходов по агентам.
- Текущая боёвка и emission escape работают через прямое обращение к `state` — их тоже нужно переключить на контекст, но это нужно делать осторожно, не ломая существующие тесты.
- `visible_entities` — нужно определить видимость. Сейчас "видят" всех co-located агентов плюс тех, о ком есть недавняя память. Нужно явно описать модель видимости.

### 3.2. Слой 2: NeedScores ⚠️ Правильная идея, но нужна формула

**Концепция верная.** Флоат-скоры (0.0..1.0) вместо жёстких порогов — это хороший подход, дающий гибкость. Wealth gate логично встроить сюда.

**Критическое замечание:** документ не описывает формулы расчёта скоров. Это необходимо сделать до реализации.

Предложение по формулам:

```python
# Примерный подход (нужно откалибровать):
survive_now     = 1.0 if hp <= 10 else max(0, (30 - hp) / 30)
heal_self       = max(0, (50 - hp) / 50)
eat             = agent["hunger"] / 100.0
drink           = agent["thirst"] / 100.0  
sleep           = agent["sleepiness"] / 100.0

# Wealth gate:
_wealth_ratio   = min(1.0, wealth / material_threshold)
get_rich        = (1 - _wealth_ratio) * 0.6 + 0.1  # материальный драйв
hunt_target     = _wealth_ratio * base_hunt_urgency   # усиливается после wealth gate
```

**Важно:** текущие пороги (hunger >= 70, thirst >= 70, hp <= 30) — это жёсткие константы. При переходе к скорам нужно убедиться, что поведение агентов не изменилось для крайних случаев. Существующие тесты помогут это проверить.

### 3.3. Слой 3: Intent ✅ Правильная идея, хорошо декомпозирован

**Сильное место спецификации.** Intent — это именно тот слой, которого не хватает. Сейчас `agent["current_goal"]` — это строка типа `"get_rich"`, `"flee_emission"`, `"get_weapon"`. Это ближе к intent'у, чем к глобальной цели, но без формальной структуры.

**Что важно:**
- Intent должен быть одним в каждый момент (это явно указано как инвариант 11.1 — верно).
- Нужен механизм **прерывания intent'а**. Сейчас emission escape и heal прерывают всё остальное — при Intent-модели нужно явно указать, какие intents прерываемы, а какие нет.
- `interruptibility` в структуре Plan (см. 4.4 спецификации) — правильный подход.

### 3.4. Слой 4: Planning ⚠️ Нужна декомпозиция шагов плана

**Идея верная, но реализация требует дополнительного описания.**

Ключевой вопрос: что является «шагом плана» по отношению к текущему `scheduled_action`?

Сейчас `scheduled_action` — это одно долгое действие (путешествие, сон, исследование). Спецификация предлагает, чтобы `schedule_action` стал «текущим исполняемым шагом плана» — это верно архитектурно, но означает что:

1. Нужна **очередь шагов плана** (например: [travel_to_trader, trade_buy_item, travel_to_anomaly]).
2. При прерывании плана (emergency) очередь нужно уметь сбросить или приостановить.
3. Текущий `scheduled_action` хранится в `agent["scheduled_action"]` как dict — нужна миграция.

**Конкретный риск:** если агент находится в mid-travel (`scheduled_action = {type: "travel", ...}`), а мы добавляем Plan сверху — нужно описать, как old-style scheduled_action отображается на Plan до завершения миграции этапа 2.

### 3.5. Слой 5: Execution ✅ Хорошо описан

Разделение ответственности (executor не принимает стратегических решений) — правильный принцип. Именно это нарушено в текущем коде: `_bot_buy_from_trader` одновременно решает «покупать ли» и «что покупать» и «как записать это в память».

**Замечание:** нужно разделить текущие функции на две группы:
- «Evaluators» — возвращают что-то без side effects
- «Executors» — только применяют уже принятое решение

Это потребует рефакторинга текущих `_bot_buy_from_trader`, `_bot_sell_to_trader`, `_bot_schedule_travel` и т.д.

## 4. Анализ социальной архитектуры

### 4.1. RelationState ✅ Хорошо спроектирован, но требует производительностного анализа

Структура RelationState (attitude, trust, fear, respect, hostility, debt, faction_bias, ...) — полноценная. Это позволит реализовать дипломатию.

**Производительный риск:** хранение RelationState для каждой пары агентов — O(N²) по памяти. При 50 агентах это 2450 объектов. При 200 агентах — ~20000. Нужно решить:
- Хранить только для агентов, с которыми было взаимодействие (ленивая инициализация)?
- Или хранить только для co-located агентов (пересчитывать при перемещении)?
- Дефолтные значения для «незнакомых» агентов.

**Рекомендация:** ленивая инициализация с дефолтом `attitude="neutral"` для всех пар без истории.

### 4.2. Диалоги ⚠️ Минимальная реализация достаточна, но нужно описать тайминг

Концепция диалога как «обмена структурированными намерениями» (без текстового дерева реплик) — правильный MVP-подход. Уже есть прецедент: `_bot_ask_colocated_stalkers_about_item` реализует прямой «вопрос-ответ» без диалогового движка.

**Что нужно уточнить:**
- Сколько тиков занимает диалог? (1 тик? несколько?)
- Может ли диалог быть прерван (выброс, атака)?
- Что происходит, если один участник умирает в процессе?
- Инвариант 11.3 (отношения меняются только через события/диалог) — верен, но нужно описать атомарность изменений.

### 4.3. Группы ⚠️ Правильная концепция, но условия роспуска не описаны

GroupState хорошо спроектирован. Условия создания группы (пункт 6.2) чёткие.

**Что отсутствует:**
- Условия роспуска группы (лидер умер? разные глобальные цели? конфликт отношений?)
- Что происходит с `shared_plan` группы, когда один член хочет уйти?
- Как группа получает сигнал о смерти участника?
- Что значит `follow_group_plan` для агента с активным emergency need (heal_self: 0.95)?

Инвариант 11.4 (группа всегда имеет лидера) — правильный, но нужен механизм выбора нового лидера при смерти текущего.

## 5. Анализ плана миграции (этапы 1–7)

### Этап 1 — Ввести новые структуры, не ломая старый код

**Оценка: НИЗКИЙ риск, ВЫСОКАЯ ценность, рекомендую начать здесь.**

Создание `AgentContext`, `NeedScores`, `Intent`, `Plan`, `RelationState`, `GroupState` как dataclass'ов (или TypedDict) без изменения существующей логики — полностью безопасно. Это даёт основу для всего остального.

```python
# Пример минимального AgentContext (для начала):
from dataclasses import dataclass, field
from typing import Optional, Dict, Any, List

@dataclass
class AgentContext:
    agent_id: str
    self_state: Dict[str, Any]
    location_state: Dict[str, Any]
    world_context: Dict[str, Any]
    # ... остальные поля добавляются постепенно
```

### Этап 2 — Завернуть текущий cascade в Intent/Plan interface

**Оценка: СРЕДНИЙ риск. Нужна осторожность.**

«Текущий каскад продолжает работать, но возвращает intent + plan» — звучит просто, но технически требует рефакторинга ~560 строк `_run_bot_action_inner`. Ключевой момент: нужно выделить момент принятия решения (что делать) от исполнения (делать).

Предлагаемая тактика:
1. Сначала добавить `agent["current_intent"]` (строка) рядом с `agent["current_goal"]` — без изменения логики.
2. Затем постепенно переносить блоки `_run_bot_action_inner` в Intent-handler'ы.
3. Существующие тесты должны оставаться зелёными на каждом шаге.

### Этап 3 — Социальная модель

**Оценка: СРЕДНИЙ риск. Новая территория.**

Нет текущей зависимости от social model → добавить изолированно несложно. Сложность в интеграции: нужно решить, где хранить RelationState (в `state["relations"]`? внутри `agent`?). 

**Рекомендация:** `state["relations"][agent_id][other_id]` — читаемо и совместимо с текущим подходом (все данные в `state`).

### Этап 4 — DialogueSession

**Оценка: СРЕДНИЙ-ВЫСОКИЙ риск. Принципиально новая механика.**

Нужно очень аккуратно интегрировать с тик-циклом. Диалог — это временное состояние взаимодействия двух агентов. Аналог есть: `combat_interactions`. Именно по образцу `_process_all_combat_interactions` можно построить `_process_all_dialogue_sessions`.

### Этап 5 — GroupState

**Оценка: ВЫСОКИЙ риск. Самая сложная механика.**

Группы требуют синхронизации принятия решений между несколькими агентами. Это фундаментально меняет тик-цикл: сейчас решения агентов независимы, с группами нужна координация.

**Рекомендация:** начать с MVP-группы из 2 агентов, без групповых нужд, только `follow_leader_travel`. Это даёт основу без взрыва сложности.

### Этапы 6–7 — Group needs, Combat + Social

**Оценка: ВЫСОКИЙ риск. Не делать до готовности этапов 1–5.**

Потребности группы и интеграция боёвки с социальной моделью — это логичное завершение, но они зависят от стабильности всех предыдущих слоёв.

## 8. Инварианты: оценка

Все 5 инвариантов из секции 11 — правильные:

- **Intent invariant** (один dominant intent) — верно и важно для determinability.
- **Plan invariant** (scheduled action = шаг плана) — верно, нужна явная миграция.
- **Social invariant** (отношения меняются только через события) — верно, нужен audit trail.
- **Group invariant** (у группы есть лидер и shared goal) — верно, добавить условие роспуска.
- **Combat invariant** (бой не живёт отдельно от social) — верно, самый сложный в реализации.

## 9. Риски и рекомендации по управлению ими

### Критический риск: Регрессия в работающей игровой логике

Текущая система хотя и монолитна, но стабильно работает и покрыта тестами (~2500+ строк тестов). Любое изменение `_run_bot_action_inner` или `tick_zone_map` несёт риск регрессии.

**Рекомендация:**
1. Перед этапом 2 запустить full test suite и зафиксировать baseline.
2. На каждом этапе миграции запускать тесты — без красных тестов не переходим дальше.
3. Не удалять старый каскадный код до тех пор, пока новый не даёт идентичный результат.

### Важный риск: Совместимость состояния (save states)

Если игра уже запущена, у агентов нет `intent`, `plan`, `relations`, `group`. Нужна one-time migration.

**Рекомендация:** использовать паттерн `if not state.get("_v2_migrated")`, уже применённый в `_terrain_migrated_v3`.

### Умеренный риск: Производительность

x100 режим = ~10 тиков/сек. При 50 агентах это 500 решений/сек. Каждое решение не должно быть дороже текущего.

**Рекомендация:** профилировать `context_builder` и `needs evaluator` на ранних этапах.
