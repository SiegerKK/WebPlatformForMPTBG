# Demo Sector — спецификация первой демонстрационной игры

## 1. Назначение

`Demo Sector` — первая демонстрационная игра на платформе.  
Её задача — не быть “интересной игрой”, а проверить, что архитектура реально работает.

Она должна доказать:

- что матчи создаются;
- что есть стратегический и тактический уровни;
- что работает переход между контекстами;
- что команды, события и проекции связаны правильно;
- что дедлайны и автоходы работают;
- что UI schema действительно позволяет отрисовывать игру;
- что генерация и replay детерминированы.

---

## 2. Концепт игры

Это простая двухуровневая игра:

### Стратегический уровень
- карта 10x10;
- два игрока;
- у каждого по 2 группы;
- группы перемещаются по клеткам;
- некоторые клетки содержат ресурсные узлы;
- если группы противников оказываются в одном секторе, начинается бой.

### Тактический уровень
- маленькая карта 8x8;
- каждая группа разворачивается в 3 юнита;
- юниты ходят по очереди;
- могут двигаться, атаковать, отступать;
- бой завершается уничтожением, отступлением или капитуляцией стороны.

## 8. Генерация матча

## 8.1. Seed
При создании матча задаётся seed.

Одинаковый seed + одинаковая версия генератора должны давать одинаковый стартовый мир.

## 8.2. Генерация стратегической карты
Генератор делает:

1. строит карту 10x10;
2. заполняет terrain типами;
3. размещает 4 ресурсных узла;
4. размещает стартовые группы игроков;
5. создаёт fog initialization.

### Упрощённые terrain types
- `plain`
- `forest`
- `ruins`

Пока terrain влияет только на UI и на тактический шаблон боя.

## 8.3. Генерация тактической карты
При старте боя:
1. выбирается tactical template;
2. строится сетка 8x8;
3. размещаются 4–8 obstacles;
4. размещаются 2–4 cover cells;
5. размещаются стартовые позиции сторон.

---

## 9. Стратегические правила

## 9.1. Ход
На стратегическом уровне — strict turn-based.

### Параметры
- один игрок активен одновременно;
- deadline на ход: например, 24 часа в production и 60 секунд в local demo;
- если игрок не походил — fallback bot завершает ход.

## 9.2. Перемещение
Команда `move_group`:
- выбирает группу;
- указывает соседнюю клетку;
- валидируется ownership и move_range;
- при успешном применении создаёт `GroupMoved`.

## 9.3. Столкновение
Если группа входит в клетку, где уже есть вражеская группа:
- создаётся событие `BattleTriggered`;
- создаётся child context `tactical_battle`;
- стратегический контекст может перейти в `suspended` для связанных сущностей.

## 9.4. Победа на стратегии
Для демо можно сделать простые условия:
- уничтожить обе группы противника;
или
- контролировать больше ресурсных узлов к N-му ходу.

---

## 10. Тактические правила

## 10.1. Развёртывание
Каждая стратегическая группа разворачивается в 3 юнита.

### Mapping
- 1 group -> 3 unit

Пока все юниты одинаковые.

## 10.2. Порядок хода
Тактический бой — strict turn-based по сторонам.

Внутри хода стороны игрок может активировать своих юнитов в любом порядке, но каждый юнит действует один раз.

## 10.3. Движение
Команда `move_unit`:
- выбрать юнита;
- указать target cell;
- путь должен быть допустим;
- клетка должна быть свободна;
- расстояние <= move_range.

## 10.4. Атака
Команда `attack_unit`:
- выбрать атакующего;
- выбрать цель;
- цель должна быть в радиусе;
- создаётся событие `AttackResolved`;
- урон применяется к hp цели;
- при hp <= 0 создаётся `UnitDestroyed`.

## 10.5. Укрытие
Если юнит стоит на `cover`, входящий урон уменьшается на 1.

## 10.6. Отступление
Команда `retreat`:
- сторона добровольно завершает бой;
- battle result = defeat/withdrawal;
- surviving units частично конвертируются обратно в group strength.

## 10.7. Завершение боя
Бой заканчивается, если:
- все юниты одной стороны уничтожены;
- сторона отступила;
- сработал hard turn limit.

---

## 11. Агрегация результата боя назад в стратегию

После завершения tactical context нужно:

1. определить победителя;
2. вычислить surviving strength;
3. обновить стратегические group entities;
4. удалить уничтоженные группы;
5. разблокировать стратегический контекст;
6. записать summary events.

### Упрощённая модель
- если уничтожены все 3 тактических юнита, стратегическая группа удаляется;
- если осталось 1–2 юнита, стратегическая группа остаётся с уменьшенным hp_pool.

## 14. Visibility и fog of war

Для демо нужна **простая, но реальная** система видимости.

## 14.1. На стратегии
Игрок видит:
- свои группы;
- соседние клетки вокруг своих групп;
- известные ранее клетки как discovered;
- вражескую группу, если она в текущей зоне видимости.

## 14.2. В тактике
Обе стороны видят всю tactical map.

Для первой демо этого достаточно.

## 16. Боты

Для демо нужен очень простой bot policy.

## 16.1. Стратегический bot
Алгоритм:
1. если рядом враг — идти в его сторону;
2. иначе идти к ближайшему ресурсному узлу;
3. если нечего делать — end turn.

## 16.2. Тактический bot
Алгоритм:
1. если враг в радиусе — атаковать;
2. иначе идти к ближайшему врагу;
3. если ход невозможен — end turn.

Это достаточно, чтобы проверить fallback logic.

## 17. Replay требования

Демо-игра обязана поддерживать:
- воспроизведение партии по командам/событиям;
- одинаковый результат при одинаковом seed и одинаковом списке команд.

## 19. Definition of Done для Demo Sector

Демо считается готовой, если:

1. локально запускается через docker-compose;
2. можно создать матч через UI или seed script;
3. два участника могут зайти в матч;
4. стратегическая карта отображается;
5. можно походить группой;
6. при столкновении запускается tactical context;
7. tactical бой можно доиграть до результата;
8. результат возвращается в strategy;
9. таймер хода и fallback bot работают;
10. replay проходит детерминированно.

---

## 20. Почему именно эта демо-игра

Потому что она проверяет почти все ключевые свойства платформы без перегрузки сложной механикой:

- два уровня контекста;
- асинхронные ходы;
- bots;
- генерация;
- UI schema;
- transitions;
- replay;
- projections.

Это идеальный полигон перед созданием более сложных игр.
