## 2. Что нужно доработать

Эти части есть в GDD, но в коде либо отсутствуют, либо реализованы не полностью.

### 2.1. 🔴 ВЫСОКИЙ ПРИОРИТЕТ: Базовые потребности персонажа (нужды)

**GDD:** §20, §11 — голод, жажда, сонливость должны расти каждый час и влиять на поведение агента.

**Что есть сейчас:**
- Поле `stamina` (0–100) — частичный аналог сонливости.
- Поля `hp`, `radiation` — есть.
- Нет отдельных `hunger`, `thirst`, `sleepiness`.

**Что нужно сделать:**
1. Добавить поля в модель агента: `hunger` (0–100), `thirst` (0–100), `sleepiness` (0–100).
2. В `zone_generator.py` — проставить стартовые значения (например, hunger=20, thirst=20, sleepiness=10).
3. В `tick_rules.py` — каждый час увеличивать нужды:
   - `hunger += 3` за час (через ~8 часов без еды станет критичным)
   - `thirst += 5` за час (вода кончается быстрее)
   - `sleepiness += 4` за час
4. При sleep — сбрасывать `sleepiness`.
5. При eat/drink (consume item) — сбрасывать `hunger` / `thirst`.
6. При значениях > 80 — накладывать штраф на HP (медленная деградация: −1 HP/час при голоде > 80).

**Файлы:** `generators/zone_generator.py`, `rules/tick_rules.py`

### 2.2. 🔴 ВЫСОКИЙ ПРИОРИТЕТ: Очередь действий агента (action queue)

**GDD:** §6.7, §17 — у агента должна быть **очередь** намерений, а не только одно текущее действие.

**Что есть сейчас:**
- Одно поле `scheduled_action` — только одна запланированная задача.

**Что нужно сделать:**
1. Добавить поле `action_queue: list[dict]` в модель агента (рядом со `scheduled_action`).
2. Формат каждого элемента очереди (по GDD §17.2):
   ```json
   {
     "action_type": "travel",
     "planned_target": "loc_bar",
     "total_hours": 4,
     "remaining_hours": 4,
     "interruptible": true,
     "created_reason": "player_command",
     "priority": 1
   }
   ```
3. В тике при завершении текущего `scheduled_action` — брать следующий элемент из `action_queue` и делать его `scheduled_action`.
4. Добавить команду `queue_action` в `zone_map` (или расширить существующие команды параметром `enqueue=true`).
5. Добавить команду `clear_queue` — сброс плана агента.

**Файлы:** `generators/zone_generator.py`, `rules/tick_rules.py`, `rules/world_rules.py`, `definition.py`

### 2.3. 🔴 ВЫСОКИЙ ПРИОРИТЕТ: Иерархия приоритетов бота (decision priorities)

**GDD:** §11 — порядок принятия решений: лечение → защита → помощь другу → еда/вода → сон → обогащение → глобальная цель.

**Что есть сейчас:**
- В `tick_rules.py` (строки 340–408): очень простая логика — HP ≤ 20 → skip, артефакт → взять, иначе случайный ход.
- `stalker_bot.py` — тоже без приоритетной иерархии.

**Что нужно сделать:**
1. В `bots/stalker_bot.py` и в inline-логике `tick_rules.py` реализовать приоритеты:
   - **P1 (лечение):** если hp < 30 и в инвентаре есть медкит → применить медкит;
   - **P2 (защита):** если на локации есть агрессивный мутант или враг → retreat/move to safe;
   - **P3 (еда/вода):** если hunger > 70 или thirst > 70 → применить консуммабл или двигаться к торговцу;
   - **P4 (сон):** если sleepiness > 75 и на safe_hub → запланировать сон;
   - **P5 (обогащение):** если артефакты на локации → взять; если в зоне богатой лутом → explore;
   - **P6 (глобальная цель):** продолжить очередь или двигаться к цели.
2. Применять один и тот же код приоритетов и для `stalker_bot`, и для `fallback_player_bot`.

**Файлы:** `bots/stalker_bot.py`, `bots/fallback_player_bot.py`, `rules/tick_rules.py`

## 3. Что нужно убрать или упростить

### 3.1. Упростить бот-логику в `tick_rules.py`

**Проблема:** в `tick_rules.py` встроена inline-логика бота (строки 340–408) — это дублирование с `stalker_bot.py`.  
**Решение:** перенести всю бот-логику в `bots/stalker_bot.py`, из `tick_rules.py` просто вызывать `StalkerBotPolicy.decide(agent, state)`.  
Это уменьшит дублирование и упростит поддержку.

### 3.2. Не развивать `location_exploration` прямо сейчас

`exploration_rules.py` реализует grid-based exploration (движение по 8 направлениям внутри локации). По GDD это "частичный" контекст — для MVP достаточно explore как scheduled_action в zone_map.  
**Решение:** оставить файл, но не добавлять новый UI для него — пусть работает как есть.

### 3.3. Мутанты: только собаки в MVP

GDD §13.2 — "первый тип мутанта: только собаки".  
**Что есть:** `balance/mutants.py` содержит `blind_dog`, `flesh`, `zombie`, `bloodsucker`, `psi_controller`.  
**Решение:** не удалять остальных — они не мешают. Но в `zone_generator.py` при создании матча спавнить **только** `blind_dog` (уже, вероятно, так и сделано — нужно проверить). Это не требует изменений кода, только проверки конфига генератора.

## 5. Что НЕ делать в MVP (из GDD §26.2)

Следующее явно исключено из первой версии и не должно попадать в ближайшие PR:

- ❌ Сложная дипломатия и репутационная система между фракциями
- ❌ Развитая фракционная война
- ❌ Выбросы (dynamic zone hazards)
- ❌ Продвинутые аномальные и артефактные эффекты (сейчас артефакты — просто ценные предметы для продажи)
- ❌ Глубокая тактическая боёвка (инициатива, укрытия, тактическая позиция)
- ❌ Групповые отряды
- ❌ Транспорт
- ❌ Полноценная quest-система (zone_event с LLM GM — достаточно)
- ❌ Особые виды мутантов сложнее собаки

## 7. Краткое резюме

**Что уже работает и не трогаем:**  
Фреймворк, тик, карта-граф, travel/explore/sleep, торговля, боевой контекст, zone_event с LLM, генерация мира, фронтенд с тремя вкладками.

**Три главных доработки для живой Зоны:**
1. **Нужды** — hunger/thirst/sleepiness деградируют, бот на них реагирует.
2. **Очередь действий** — агент ведёт план, а не одно текущее действие.
3. **Encounter trigger** — пересечение сталкера с мутантом автоматически запускает бой.

Без этих трёх вещей игра остаётся демо. С ними — живая симуляция, на которой можно строить дальше.
