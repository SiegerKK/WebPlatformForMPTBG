# Вопросы по NPC Brain v3 — Раунд 5

> Контекст: прочитал `docs/npc_v3_questions_round4_answer.md`.
> Большинство решений для PR 1 уже зафиксированы, но осталось несколько точек, где ответ всё ещё допускает два допустимых варианта scope/контракта.

---

## 1. `brain_trace` coverage: полный критерий уже в PR 1 или phased rollout?

В ответе раунда 4 целевой критерий сформулирован жёстко:

```text
После tick_zone_map у каждого живого bot stalker должен быть brain_trace за текущий тик.
```

Но ниже добавлен компромисс:

```text
Если сложно покрыть все ранние return сразу,
допускается временный критерий PR 1.0 / PR 1.1.
```

Вопрос: какой scope считать **официальным acceptance для PR 1**?

- Вариант A: сразу требуем полный coverage для всех живых bot stalkers;
- Вариант B: в PR 1 покрываем только `scheduled_action` + `_run_bot_decision_v2_inner`, а ранние legacy-return переносим в PR 1.1.

Это нужно зафиксировать до написания тестов, иначе acceptance получится двусмысленным.

---

## 2. `brain_trace` enums: фиксируем только реально используемые значения PR 1 или сразу future-safe набор?

Раунд 4 предлагает стабильный TS-контракт, но он уже включает значения, которые в PR 1 ещё не будут реально генерироваться:

- `pause`
- `adapt`
- `scheduled_action`
- `system`
- `complete`

Вопрос: что именно считается стабильным публичным контрактом PR 1?

- Вариант A: фиксируем **только** значения, которые реально эмитятся в PR 1;
- Вариант B: фиксируем расширенный future-safe enum заранее, даже если часть значений пока не используется.

Это важно для фронта и тестов snapshot/typing.

---

## 3. `_v3_replanned_after_monitor_turn`: обязательно ли чистить в конце тика?

В ответе раунда 4 предложен guard:

```python
agent["_v3_replanned_after_monitor_turn"] = world_turn
```

и затем сказано, что поле:

- можно оставить до следующего тика и просто перезаписывать;
- или чистить в конце `tick_zone_map`.

Вопрос: какой вариант принять как норму PR 1?

- A) transient field обязан удаляться в конце тика;
- B) поле можно не чистить, если проверка всё равно завязана на `world_turn`.

Это небольшая деталь, но от неё зависит чистота state и предсказуемость тестов/сериализации.

---

## 4. `active_plan_v3` в API PR 1: показываем как часть стабильного frontend-contract или оставляем purely optional debug field?

Раунд 4 фиксирует `active_plan_v3` как debug/retrofit metadata и одновременно предлагает `brain_trace.active_plan?: unknown`.

Вопрос: нужно ли уже в PR 1 считать `active_plan_v3`/`brain_trace.active_plan` частью публичного UI-контракта?

- Вариант A: да, сразу описать минимальный TS-тип и начать показывать на фронте;
- Вариант B: нет, в PR 1 это backend/debug metadata, фронт опирается только на `brain_trace.current_thought` и `events`.

Без этого непонятно, надо ли тестировать форму `active_plan_v3` как стабильную.

## 5. `plan_monitor_aborted_action`: optional payload нужен ли в контракте уже сейчас?

Раунд 4 фиксирует обязательные поля:

- `agent_id`
- `scheduled_action_type`
- `reason`
- `dominant_pressure`

И отдельно предлагает optional-поля:

- `cancelled_target`
- `cancelled_final_target`
- `current_location_id`
- `turns_remaining`

Вопрос: optional-поля уже стоит зафиксировать как “если есть — в таком формате”, или в PR 1 они полностью вне контракта?

- Вариант A: сразу закрепить точные имена и типы optional-полей;
- Вариант B: в PR 1 гарантировать только обязательный минимум, остальное не обещать.

Это поможет не получить later breaking change в event consumers.

---

## 6. Memory dedup: throttle применяется только к `abort`, или ко всем будущим monitor-memory event'ам?

В раунде 4 throttle явно описан для:

```text
plan_monitor_action_aborted
```

Но если в следующих PR появятся `pause/adapt/completed`, логика дедупликации может понадобиться шире.

Вопрос: в PR 1 нужно сразу сделать dedup helper общего назначения или оставить узкоспециализированный guard только для `abort`?

- A) общий helper для всех plan-monitor memory events;
- B) только `abort`, остальное не обобщать заранее.

Это влияет на дизайн helper'а и на то, насколько широко его стоит выносить уже сейчас.
