## Типы записей

| Тип | Назначение |
|-----|------------|
| `"observation"` | Факт о мире, замеченный агентом (не его собственное решение). Используется для блокировок. |
| `"decision"` | Решение, принятое ботом (объяснение выбора действия). |
| `"action"` | Результат совершённого действия (артефакт поднят, предмет куплен, ...). |

## Как работают блокировки

Два механизма блокировки основаны на `"observation"` записях:

### 1. Блокировка поиска предметов — `_item_not_found_locations()`

**Функция:** `_item_not_found_locations(agent, item_types)` → `set[str]`  
**Файл:** `tick_rules.py`, около строки 1715

Возвращает множество `location_id`, куда агент **не должен идти** за предметами категории `item_types`, потому что он уже там был и ничего не нашёл.

**Логика снятия блокировки:** Блок снимается автоматически, если позже (бо́льший `world_turn`) появилась запись `observed:items` для тех же типов предметов в той же локации (кто-то бросил предмет на землю).

**Используется в:** `_find_item_memory_location()` — функция поиска ближайшей локации с нужными предметами (исключает заблокированные).

---

### 2. Блокировка поиска артефактов — `_confirmed_empty_locations()`

**Функция:** `_confirmed_empty_locations(agent)` → `frozenset[str]`  
**Файл:** `tick_rules.py`, около строки 878

Возвращает множество `location_id`, которые агент **не должен исследовать** повторно — он уже искал артефакты и не нашёл ничего.

**Логика снятия блокировки:** Снимается после выброса. Когда выброс заканчивается, каждый живой агент получает запись `emission_ended`. Все записи `explore_confirmed_empty`, сделанные **до** этой записи, считаются устаревшими: выброс порождает новые артефакты, и агент снова готов исследовать зоны.

**Используется в:**
- `_bot_gather_resources()` — фаза G2/G3: пропускает подтверждённо-пустые зоны при выборе аномалии для исследования.
- `_bot_pursue_goal()` — аналогично для цели `get_rich`.

## Полный справочник по `action_kind`

---

### Наблюдения (`"observation"`)

#### `death`
| Контекст | Условие |
|---------|---------|
| `tick_zone_map()` | HP ≤ 0 от голода или жажды (проверка на границе часа) |
| `tick_zone_map()` | Нахождение на опасной местности в момент начала выброса |
| `_process_scheduled_action()` | HP ≤ 0 от урона аномалии во время прыжка (travel hop) |
| `_resolve_exploration()` | HP ≤ 0 от аномалии во время разведки (вероятность 15% × anomaly_activity/10) |

Запись: тип `"observation"`, текст «💀 Смерть».  
Поля `effects`: `action_kind`, `cause` (`"starvation"` / `"thirst"` / `"emission"` / `"anomaly_travel"` / `"anomaly_exploration"`), `location_id`, при необходимости `terrain` или `anomaly_type`.

---

#### `emission_imminent`
Запись за 10–15 ходов до начала выброса. Записывается **один раз** за цикл, смещение (offset) кешируется в `state["emission_warning_offset"]`.

Поля `effects`: `action_kind`, `emission_scheduled_turn`, `turns_until`.  
Текст: «⚠️ Скоро выброс!»

Используется: бот проверяет наличие этой записи в памяти, чтобы принять решение о бегстве в укрытие (`flee_emission`, `wait_in_shelter`).

---

#### `emission_started`
Записывается всем живым агентам в момент начала выброса (`world_turn == emission_scheduled_turn`).

Поля `effects`: `action_kind`.  
Текст: «⚡ Начался выброс!»

---

#### `emission_ended`
Записывается всем живым агентам по окончании выброса (`world_turn >= emission_ends_turn`).

Поля `effects`: `action_kind`.  
Текст: «✅ Выброс закончился»

**Важно:** Именно эта запись инвалидирует все старые `explore_confirmed_empty` (см. выше).

---

#### `explore_confirmed_empty`
Записывается при **любом** неудачном поиске артефактов в аномальной зоне.

Два случая, оба дают одинаковую запись:
1. **Зона пустая** — на локации нет ни одного артефакта.
2. **Промах** — артефакты есть, но бросок 50% не удался.

Поля `effects`: `action_kind`, `location_id`.  
Тип памяти: `"observation"`.  
Текст: «Аномалия в «{loc_name}» не дала артефакт»  
Summary: «Обыскал «{loc_name}» — артефактов не нашёл.»

**Снимается:** при появлении записи `emission_ended` с `world_turn` > `world_turn` данной записи.

> ℹ️ До текущей версии существовала отдельная запись `explore_miss` типа `"decision"` для случая промаха — она была удалена. Теперь оба исхода записываются единообразно как `explore_confirmed_empty`.

---

#### `item_not_found_here`

Подробный разбор — в разделе [«Детально: `item_not_found_here`»](#детально-item_not_found_here).

---

#### `observed:stalkers`
Когда в локации появились другие сталкеры или торговцы (дедупликация по содержимому).

Поля `effects`: `observed: "stalkers"`, `location_id`, `names: List[str]`.  
Текст: «Вижу персонажей в «{loc_name}»»

---

#### `observed:mutants`
Когда в локации есть живые мутанты (дедупликация).

Поля `effects`: `observed: "mutants"`, `location_id`, `names: List[str]`.  
Текст: «Вижу мутантов в «{loc_name}»»

---

#### `observed:items`
Когда на земле в локации есть предметы (дедупликация по составу).

Поля `effects`: `observed: "items"`, `location_id`, `item_types: List[str]`.  
Текст: «Вижу предметы в «{loc_name}»»

**Важно:** Именно эта запись **снимает** блокировку `item_not_found_here`, если появилась позже неё.

### Решения (`"decision"`)

#### `exploration_interrupted`
Агент прерывает идущую разведку аномалии из-за угрозы выброса.

Условие: `scheduled_action.type == "explore_anomaly_location"` AND `turns_remaining > 0` AND выброс активен/близок AND не `emergency_flee`.

Поля `effects`: `action_kind`, `reason: "emission_warning"`, `location_id`.  
Текст: «⚡ Прерываю исследование из-за выброса»

---

#### `travel_interrupted`
Агент прерывает движение из-за угрозы выброса. Два случая:

1. **В пути (один прыжок):** `turns_remaining > 0` во время хода.
2. **Между прыжками:** есть `remaining_route` (многоточечный маршрут), очередной прыжок ещё не начат.

Поля `effects`: `action_kind`, `reason: "emission_warning"`, `location_id`, `cancelled_target`.  
Текст: «⚡ Прерываю движение из-за выброса»

---

#### `route_changed`
Следующий запланированный прыжок в маршруте стал недоступен, но BFS нашёл альтернативный маршрут до конечной цели.

Поля `effects`: `action_kind`, `old_next`, `new_next`, `final_target`.  
Текст: «Смена маршрута из-за недоступности перехода»

---

#### `goal_cancelled`
Следующий прыжок недоступен И BFS не нашёл альтернативы к `final_target_id`.

Поля `effects`: `action_kind`, `cancelled_target`.  
Текст: «Смена решения из-за недоступности цели»

---

#### `flee_emission`
Бот решил бежать в укрытие от выброса.

Условие: нахождение на опасной местности (`plain`, `hills`, `swamp`, `field_camp`, `slag_heaps`) AND (выброс активен OR `emission_imminent` в памяти).

Поля `effects`: `action_kind`, `destination`, `terrain`.  
Текст: «⚡ Бегу от выброса!»

---

#### `wait_in_shelter`
Бот решил ждать окончания выброса в текущей безопасной локации.

Условие: выброс активен/близок AND NOT опасная местность. Записывается не более одного раза подряд (дедупликация в рамках одного дерева решений).

Поля `effects`: `action_kind`, `location_id`.  
Текст: «🛡️ Жду в укрытии»

---

#### `sleep_decision`
Бот решил поспать.

Условие: `sleepiness >= 75`.

Поля `effects`: `action_kind`, `hours`.  
Текст: «Ложусь спать»

---

#### `seek_item`
Бот решил идти к локации за конкретным предметом (подобрать с земли). Различные `item_category`:

| Категория | Условие |
|-----------|---------|
| `"medical"` (экстренный) | HP ≤ 30, нет аптечки в инвентаре, торговец достижим/доступен по цене |
| `"food"` (экстренный) | hunger ≥ 70, нет еды, торговец достижим |
| `"drink"` (экстренный) | thirst ≥ 70, нет питья, торговец достижим |
| `"weapon"` | Нет экипированного оружия, предмет нашёлся в памяти, не текущая локация |
| `"armor"` | Нет экипированной брони, предмет нашёлся в памяти |
| `"ammo"` | Оружие требует патроны, нет в инвентаре, нашлось в памяти |
| `"medical"` | Нет аптечки, нашлась в памяти |
| `"food"` | Нет еды, голод > 30, нашлась в памяти |
| `"drink"` | Нет питья, жажда > 30, нашлось в памяти |

Поля `effects`: `action_kind`, `destination`, `item_category`, `item_types`.

#### `buy_item`
Бот решил идти к торговцу за покупкой (не за подбором с земли).

| Категория | Условие |
|-----------|---------|

| `"armor"` | Нет брони, аналогично |
| `"ammo"` | Нет патронов для оружия, аналогично |

Поля `effects`: `action_kind`, `destination`, `item_category`.

## Детально: `item_not_found_here`

### Что это

Запись-наблюдение, которая **блокирует** повторные поездки агента за предметом в эту локацию.

### Два источника записи

#### Источник 1: `"source": "pickup"` — после подбора последнего предмета

**Функция:** `_bot_pickup_item_from_ground()`, около строки 1699  
**Когда:** Агент успешно поднял предмет с земли, и после этого на земле **не осталось** ни одного предмета нужной категории в этой локации.

**Эффекты:**
```json
{
  "action_kind": "item_not_found_here",
  "source": "pickup",
  "location_id": "<id>",
  "item_types": ["<type1>", "<type2>", ...]
}
```

Текст: «📭 Предметы закончились в «{loc_name}»»  
Summary: «Подобрал последний предмет нужного типа в «{loc_name}». Здесь больше нет.»

---

#### Источник 2: `"source": "arrival"` — агент пришёл, но ничего не нашёл

**Функция:** `_maybe_record_item_not_found()`, около строки 1808  
**Когда:** Агент прибыл в локацию (travel_arrived), цель поездки — поднять предмет (`seek_item` decision с `destination == loc_id`), но предметов нужного типа на земле нет.

**Условия для записи:**
1. В памяти агента есть запись `decision` с `action_kind == "seek_item"` и `destination == loc_id` и `item_category == item_category`.
2. Последнее релевантное решение НЕ было `buy_item` (торговцы не держат предметы на земле — «нет предметов» ожидаемо).
3. В этот же ход ещё не было аналогичной записи (дедупликация).

**Эффекты:**
```json
{
  "action_kind": "item_not_found_here",
  "source": "arrival",
  "location_id": "<id>",
  "item_types": ["<type1>", "<type2>", ...]
}
```

Текст: «⚠️ Предмет исчез из «{loc_name}»»  
Summary: «Пришёл в «{loc_name}» за {item_category}, но предмет уже забрали.»

---

### Когда блокировка снимается

Блок снимается, когда в памяти агента появляется запись **`observed:items`** с:
- тем же `location_id`
- пересечением `item_types`
- **более поздним** `world_turn` (т. е. запись наблюдения новее блокировки)

Это происходит, когда агент (или кто-то другой) видит предмет на земле в этой локации — `_write_location_observations()` автоматически записывает `observed:items`.

---

### Как это используется

**Функция:** `_item_not_found_locations(agent, item_types)` → `set[str]`

1. Перебирает все `"observation"` в памяти агента.
2. Собирает `last_not_found_turn[loc_id]` — последний ход, когда было записано `item_not_found_here` для данных типов предметов.
3. Собирает `last_item_obs_turn[loc_id]` — последний ход `observed:items` для тех же типов.
4. Локация **заблокирована**, если `last_not_found_turn[loc_id] >= last_item_obs_turn.get(loc_id, 0)`.

Результат передаётся в `_find_item_memory_location()`, которая исключает заблокированные локации из кандидатов для поездки.

## Детально: `explore_confirmed_empty`

### Что это

Запись-наблюдение, которая **блокирует** повторную разведку аномальной зоны агентом.

### Когда записывается

**Функция:** `_resolve_exploration()`, около строки 695  
**Условие:** Разведка завершилась без получения артефакта. Два подслучая, оба дают одинаковую запись:

| Подслучай | Описание |
|----------|----------|
| **Зона пустая** | `existing_artifacts == []` — артефактов на локации нет вообще |
| **Промах** | Артефакты есть, но бросок `rng.random() < 0.5` не удался |

**Эффекты:**
```json
{
  "action_kind": "explore_confirmed_empty",
  "location_id": "<id>"
}
```

Тип: `"observation"`  
Текст: «Аномалия в «{loc_name}» не дала артефакт»  
Summary: «Обыскал «{loc_name}» — артефактов не нашёл.»

---

### Когда блокировка снимается

После окончания выброса. Алгоритм в `_confirmed_empty_locations()`:

1. Находит максимальный `world_turn` записи `emission_ended` в памяти агента.
2. Считает запись `explore_confirmed_empty` **действительной** только если её `world_turn > last_emission_ended_turn`.
3. Записи, сделанные до последнего выброса, игнорируются — выброс порождает новые артефакты.

---

### Как это используется

Функция `_confirmed_empty_locations(agent)` → `frozenset[str]` вызывается:
- в `_bot_gather_resources()` — при выборе аномалии для разведки (фазы G2 и G3)
- в `_bot_pursue_goal()` — при поиске аномалии для цели `get_rich`

Локации из этого множества исключаются из списка кандидатов для исследования.

## Память торговцев

Торговцы имеют свою отдельную память (`trader["memory"]`), заполняемую через `_add_trader_memory()`. Лимит тот же: `MAX_AGENT_MEMORY = 2000`.

### `trade_buy` (в памяти торговца)
Торговец записывает покупку артефактов у агента.

Поля `effects`: `action_kind: "trade_buy"`, `agent_id`, `artifact_types`, `total_price`.  
Текст: «Купил {items} у сталкера {name}»

### `trade_sale` (в памяти торговца)
Торговец записывает продажу предмета агенту.

Поля `effects`: `action_kind: "trade_sale"`, `agent_id`, `item_type`, `price`.  
Текст: «Продал «{item}» сталкеру {name}»
